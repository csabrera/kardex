### 1.2 Patrones y Principios Aplicados

| Capa | Patrones |
|------|----------|
| **Backend** | Arquitectura Modular (NestJS), Clean Architecture light, Repository Pattern (Prisma), DTOs + ValidaciГіn, Guards/Interceptors, CQRS ligero en mГіdulos crГӯticos, WebSocket Gateway para eventos en tiempo real |
| **Frontend** | Feature-Based Architecture, Server Components por defecto + Client Components cuando sea necesario, SeparaciГіn UI / lГіgica de negocio / data-fetching, estado global hГӯbrido (Zustand + TanStack Query + Socket.IO) |
| **Seguridad** | RBAC (Role-Based Access Control) con permisos granulares, JWT + Refresh Tokens httpOnly, bcrypt para passwords, Helmet, CORS estricto, Rate limiting, validaciГіn de formato por tipo de documento |
| **Datos** | Soft deletes, AuditorГӯa automГЎtica, Transacciones ACID en movimientos, ГҚndices compuestos en tablas de alta lectura |
| **Tiempo Real** | Estrategia hГӯbrida: polling de TanStack Query para datos generales + WebSockets (Socket.IO) para eventos crГӯticos (alertas, transferencias, requisiciones, mantenimientos, cambios de stock en almacГ©n activo) |

### 1.3 Decisiones TГ©cnicas Justificadas

- **Monorepo con Turborepo:** pipeline de build inteligente, cachГ© remoto, excelente DX con Next.js y NestJS compartiendo tipos. MГЎs simple que Nx para este tamaГұo.
- **Prisma sobre TypeORM:** mejor DX, migraciones declarativas, tipado fuerte end-to-end, mejor comunidad actual.
- **App Router de Next.js:** Server Components reducen JS en cliente, mejor performance inicial, streaming nativo.
- **shadcn/ui:** componentes copiables (no dependencia), totalmente personalizables, basados en Radix UI (accesibilidad).
- **TanStack Query:** manejo de cachГ© de servidor, invalidaciГіn quirГәrgica, optimistic updates imprescindibles en inventario.
- **Zustand:** estado cliente liviano para UI (sidebar, modales, filtros locales) sin el boilerplate de Redux.
- **Socket.IO (NestJS Gateway + socket.io-client):** tiempo real con reconexiГіn automГЎtica, rooms por usuario/almacГ©n, fallback a long-polling si WebSocket falla.
- **Railway:** despliegue simple con PostgreSQL managed, sin configurar reverse proxy (Railway lo maneja). Ideal para MVP/presentaciГіn.
- **Recharts + Tremor:** grГЎficas responsivas, componentes de dashboard listos, estГЎndar de facto en React.
- **TanStack Table v8:** tablas headless con control total (paginaciГіn, sort, filtros, virtualizaciГіn).
- **Sonner:** sistema de notificaciones toast moderno y accesible.
- **ExcelJS + Puppeteer:** ExcelJS para exportaciГіn a hojas de cГЎlculo con estilos; Puppeteer para PDFs con layouts complejos (logo, cabeceras/pies, grГЎficas embebidas).
- **date-fns + date-fns-tz:** manejo de fechas ligero, tree-shakeable, con soporte de zonas horarias.

5. **Progressive disclosure:** mostrar lo esencial por defecto, expandir detalles bajo demanda.
6. **Accesibilidad AA:** contraste mГӯnimo 4.5:1, navegaciГіn por teclado, `aria-labels`, focus-ring visible.
7. **Responsive-first:** diseГұo mГіvil (вүҘ360px), tablet (вүҘ768px), desktop (вүҘ1024px), wide (вүҘ1440px).

### 12.2 Identidad del Usuario

- La identidad NO se hace por email (email es opcional).
- Clave Гәnica compuesta: **`(documentType, documentNumber)`**.
- En la tabla `users` hay un `UNIQUE INDEX` sobre esas dos columnas.
- Dos usuarios pueden compartir el mismo nГәmero si tienen distinto tipo (ej. un DNI `12345678` y un Pasaporte `12345678`).

### 12.6 Protecciones de Seguridad

- El password del seed se hashea con bcrypt antes de guardarlo (nunca se guarda en texto plano, ni siquiera temporalmente).
- En Railway, las variables `ADMIN_*` se pueden borrar despuГ©s del primer deploy exitoso.
